# 수다쟁이 모자 다음 단계 구현 계획

## 1. 현재 구현 상태 요약

현재 코드베이스에는 아래 항목이 반영되어 있다.

- `ChatterHat` 아이템 등록 및 헬멧 슬롯 장착 가능 설정
- `/hat say`, `/hat status`, `/hat memory dump`, `/hat memory clear`, `/hat test ollama`
- `/hat config language`, `/hat config persona`, `/hat config model`
- Ollama `/api/chat` 비동기 호출
- 최근 대화 버퍼 저장
- 월드별 메모리 JSON 저장
- 저체력 / 저허기 / 첫 밤 / 근처 몹 규칙 기반 경고
- 한국어 / 영어 / 자동 언어 설정의 기본 구조

즉, 현재는 **문서상의 Phase 1과 Phase 2의 일부가 연결된 초기 MVP** 상태다.

---

## 2. 다음 단계의 목표

다음 단계의 핵심 목표는 아래 3가지다.

1. 지금 있는 명령 기반 MVP를 실제 플레이 흐름에 더 자연스럽게 연결한다.
2. 메모리를 "저장" 수준에서 "기억하는 느낌" 수준으로 끌어올린다.
3. 설정을 코드 수정 없이 조절할 수 있도록 UI와 디버깅 도구를 추가한다.

한 줄로 정리하면:

**"지금의 작동하는 골격을, 실제로 계속 쓰고 싶은 동료 모자 경험으로 발전시키는 단계"**

---

## 3. 우선순위 기준

다음 단계는 아래 우선순위로 진행한다.

### P1. 플레이 경험에 바로 영향을 주는 것
- 응답 품질 안정화
- 이벤트 반응 품질 개선
- 과도한 발화 억제
- 기억 반영 체감 강화

### P2. 설정 편의성
- 인게임 설정 화면
- 연결 테스트 화면
- 페르소나 선택 UI

### P3. 확장성
- provider 추상화 강화
- memory 구조 고도화
- 탐험/월드 이벤트 확장

---

## 4. 구현 단계 제안

## Step 1. 대화 품질과 상태 제어 보강

### 목표
현재 대화 루프를 더 안정적이고 예측 가능하게 만든다.

### 작업
- `DialogueType`별 프롬프트 차별화 강화
- 위험 상황에서 응답 길이 강제 축약
- Ollama 응답 실패 시 fallback 문구 개선
- 중복 요청 방지와 요청 중 상태 표시
- 최근 같은 응답 반복 감지
- 플레이어가 모자를 벗었을 때 대기 중 응답 무시 처리

### 완료 기준
- 같은 상황에서 지나치게 비슷한 응답이 반복되지 않음
- 전투/위험 상황에서 긴 대사가 거의 나오지 않음
- Ollama 실패 시에도 어색하지 않은 최소 반응 유지

### 후보 클래스
- `ResponsePostProcessor` 확장
- `HatSpeechController` 발화 상태 관리 강화
- `LlmRateLimiter`
- `LlmFallbackPolicy`

---

## Step 2. 메모리 시스템 2차 구현

### 목표
최근 대화 저장을 넘어서, 플레이어의 목표와 선호를 반영한 응답을 만든다.

### 작업
- `facts` 구조를 문자열 리스트에서 객체형으로 확장
- fact 후보 추출 규칙 추가
- 목표 / 선호 / 자주 하는 행동 / 주요 사건 저장
- 오래된 recent messages를 요약 summary로 압축
- 프롬프트에 넣을 memory retrieval 전략 추가

### 예시 fact
- "플레이어는 철과 다이아 채굴을 목표로 삼고 있음"
- "플레이어는 밤 탐험을 싫어함"
- "플레이어는 실용적인 조언을 선호함"

### 완료 기준
- 몇 번 대화 후 모자가 플레이어 목표를 반영한 답변을 함
- 메모리 파일이 너무 빠르게 비대해지지 않음
- 요약 메모리가 프롬프트에 실제 반영됨

### 후보 클래스
- `FactMemoryStore`
- `MemoryFact`
- `MemoryExtractor`
- `SummaryMemoryStore`
- `MemoryRetrievalService`

---

## Step 3. 이벤트 감지 확장

### 목표
현재의 단순 경고에서, "세계와 함께 체험하는 동료" 느낌을 강화한다.

### 작업
- 첫 스폰 이벤트
- 사망 직후 반응
- 새 바이옴 진입 감지
- 장시간 침묵 후 짧은 idle banter
- 광물 발견 또는 구조물 접근 같은 탐험 이벤트 후보 추가
- 이벤트별 개별 쿨다운 / 1회성 발화 / biome별 중복 억제 추가

### 구현 원칙
- 즉시성 필요한 경고는 로컬 템플릿 우선
- 분위기형 코멘트만 선택적으로 LLM 호출
- noisy event는 rule engine에서 정제

### 완료 기준
- 저체력/적대몹 경고 외에도 탐험 맥락 반응이 생김
- 같은 바이옴/같은 상황에서 반복적으로 떠들지 않음

### 후보 클래스
- `LifecycleDetector`
- `ExplorationDetector`
- `HatEvent`
- `HatEventType`
- `EventRuleEngine`

---

## Step 4. 인게임 설정 UI

### 목표
명령어 없이 주요 설정을 조절할 수 있게 만든다.

### 작업
- 기본 설정 화면 `HatConfigScreen`
- Ollama 연결 테스트 버튼
- 모델명 입력 필드
- 언어 선택 토글
- 페르소나 프리셋 선택
- 수다 빈도 / 응답 길이 / 경고 언어 설정
- 메모리 요약 보기 / 메모리 초기화 버튼

### UI 범위
- 초기에는 단순한 vanilla `Screen` 기반
- 보기 좋은 연출보다 조작 가능성과 상태 표시 우선

### 완료 기준
- 주요 설정을 게임 안에서 변경 가능
- 연결 성공/실패가 명확히 보임
- 설정값이 `config/chatterhat.json`에 안정적으로 반영됨

---

## Step 5. 페르소나 고도화

### 목표
모자가 단순히 답하는 도구가 아니라 캐릭터처럼 느껴지게 만든다.

### 작업
- 현재 persona preset을 실제 prompt template과 더 깊게 연결
- preset별 말투 차이 강화
- custom persona text 입력 지원
- 친밀도/농담 비율/실용성 비율 같은 파라미터 추가 검토

### 완료 기준
- 페르소나를 바꾸면 응답 성격 차이가 분명하게 체감됨
- 언어 설정과 페르소나 설정이 충돌하지 않음

---

## Step 6. 디버깅과 운영 도구 보강

### 목표
왜 말했는지, 왜 침묵했는지 추적 가능하게 만든다.

### 작업
- debug 플래그 추가
- 마지막 프롬프트 미리보기 명령
- 마지막 Ollama 응답 시간 기록
- 마지막 차단된 이벤트 이유 기록
- `/hat debug event <type>` 수동 트리거
- `/hat debug prompt`

### 완료 기준
- 이벤트/발화 문제를 로그와 명령으로 재현 가능
- 언어 정책이나 메모리 반영 문제를 빠르게 추적 가능

---

## 5. 권장 구현 순서

실제 작업 순서는 아래를 권장한다.

1. Step 1 대화 품질과 상태 제어 보강
2. Step 2 메모리 시스템 2차 구현
3. Step 3 이벤트 감지 확장
4. Step 6 디버깅 도구 보강
5. Step 4 인게임 설정 UI
6. Step 5 페르소나 고도화

이 순서를 추천하는 이유:

- UI보다 먼저 내부 품질과 기억 구조를 안정화해야 한다.
- 메모리와 이벤트가 정리되어야 UI도 제대로 설계된다.
- 디버깅 도구가 먼저 있으면 이후 UI와 이벤트 작업 속도가 빨라진다.

- 요청 중복 방지 로직

- summary 생성/저장 코드

### Step 3 산출물

- 이벤트 dedupe/cooldown 관리 구조

### Step 4 산출물

- Ollama 연결 테스트 UI

- 강화된 persona preset
- custom persona 지원

### Step 6 산출물
- debug 명령 세트
- 로그 포맷 개선
- latency/발화 추적 정보

## 7. 기술 리스크와 대응

## 7.1 Ollama 지연 시간

### 리스크
- 응답이 늦으면 대화 흐름이 끊긴다.

### 대응
- 위험 경고는 로컬 템플릿 유지
- idle/comment류만 선택적으로 LLM 호출
- 대기 중 추가 호출 제한

## 7.2 메모리 비대화

### 리스크
- recent/facts/summary가 무한정 커질 수 있다.

### 대응
- recent 개수 제한
- facts 중요도 기준 pruning
- summary 주기적 재생성

## 7.3 언어 일관성 붕괴

### 리스크
- `auto` 모드에서 언어가 흔들릴 수 있다.

### 대응
- 최근 플레이어 발화 언어 우선
- 마지막 모자 응답 언어 보조 사용
- 경고 문구는 로컬 템플릿 유지

## 7.4 이벤트 과민 반응

### 리스크
- 모자가 너무 자주 말해 피로감을 준다.

### 대응
- 개별 쿨다운
- 저중요 이벤트 침묵 허용
- 최근 발화 직후에는 저우선순위 이벤트 차단

---

## 8. 테스트 체크리스트

다음 단계 구현 후 최소한 아래는 반복 테스트한다.

### 대화
- 모자 착용 시 `/hat say 안녕` 응답
- 모자 미착용 시 `/hat say` 차단
- Ollama 중단 상태에서 fallback 응답 확인

### 언어
- `ko`, `en`, `auto` 각각에서 응답 언어 확인
- 경고 문구 언어가 설정과 일치하는지 확인

### 메모리
- 최근 대화가 저장되는지 확인
- 목표/선호 fact가 누적되는지 확인
- 월드 재접속 후 memory 복원 확인

### 이벤트
- 저체력 경고 쿨다운 확인
- 첫 밤 1회성 발화 확인
- 몹 접근 시 반복 억제 확인
- 새 바이옴 진입 반응 확인

### UI
- 설정 변경 즉시 반영 여부
- 연결 테스트 성공/실패 문구 확인
- 메모리 초기화 버튼 동작 확인

## 9. 바로 다음 작업 추천

가장 먼저 착수할 작업은 아래다.

### 추천 1순위
**Step 2 메모리 시스템 2차 구현**

이유:

- 지금 MVP에서 가장 부족한 체감 포인트가 "기억한다"는 느낌이다.
- 메모리가 좋아지면 대화 품질과 캐릭터성이 함께 상승한다.
- 이후 UI를 만들어도 보여줄 가치가 있는 데이터가 생긴다.

### 추천 2순위
**Step 1 대화 품질과 상태 제어 보강**

이유:

- 현재 구조는 이미 작동하므로, 안정성 개선 효과가 빠르게 나타난다.
- 발화 중복과 긴 응답 문제를 먼저 줄이면 이후 모든 단계 품질이 올라간다.

---

## 10. 다음 구현 턴에서의 실제 시작점

다음 구현 턴에서는 아래 순서로 진행하면 된다.

1. `memory/` 패키지에 `MemoryFact`, `FactMemoryStore`, `MemoryExtractor` 추가
2. `WorldMemoryData`를 구조화된 fact/summary 포맷으로 변경
3. `DialogueCoordinator`에서 memory retrieval 주입
4. `/hat memory dump` 출력을 구조화 형식으로 개선
5. fact 생성 규칙 2~3개부터 적용

이렇게 하면 작은 범위로 시작하면서도 체감 개선이 크다.
